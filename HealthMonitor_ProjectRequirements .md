# 1. Project Overview {#project-overview}

This project is a wearable IoT health monitoring system that continuously reads biometric data from body sensors attached to an ESP32 microcontroller, streams the data to Firebase, displays live readings and insights in a Flutter mobile application, and uses a Python ML service to generate medically-grounded health insights.

## 1.1 Goals {#goals}

- Real-time monitoring of heart rate, blood oxygen (SpO2), body temperature, and physical activity.

- Structured data storage: 1-minute averages aggregated in Firebase Realtime DB (RTDB) with 24-hour rolling history.

- Intelligent health insights generated by a Python ML service using medically correct thresholds (running every 8s).

- Sleep mode detection with dedicated analysis for resting biometrics.

- End-of-day graph reports visualised inside the Flutter app.

- Scalable Firebase architecture supporting multi-user deployment.

## 1.2 What Was Already Built {#what-was-already-built}

- ESP32 firmware reading MAX30102 IR data via raw I2C registers (no library).

- Firebase Realtime DB integration for live MAX30102 readings.

- Flutter app connected to Realtime DB showing live MAX30102 values.

*Everything below describes what is being added and expanded in this version.*

# 2. Hardware {#hardware}

## 2.1 Component List {#component-list}

|                  |                                                                    |
|------------------|--------------------------------------------------------------------|
| **Component**    | **Role / Notes**                                                   |
| ESP32 Dev Module | Main MCU --- WiFi, I2C master, ADC for thermistor                  |
| MAX30102         | Heart rate & SpO2 --- I2C address 0x57, raw register access        |
| MPU6050          | Accelerometer + Gyroscope --- I2C address 0x68, activity detection |
| NTC Thermistor   | Body temperature --- analog on GPIO 34                             |
| USB Cable (data) | Must be data-capable, not charge-only                              |

## 2.2 Wiring {#wiring}

|               |                            |                                    |
|---------------|----------------------------|------------------------------------|
| **ESP32 Pin** | **Connects To**            | **Notes**                          |
| GPIO 21 (SDA) | MAX30102 SDA + MPU6050 SDA | Shared I2C bus                     |
| GPIO 22 (SCL) | MAX30102 SCL + MPU6050 SCL | Shared I2C bus                     |
| 3.3 V         | MAX30102 VIN + MPU6050 VCC | Do NOT use 5 V for MAX30102        |
| GND           | All sensor GND pins        | Common ground                      |
| GPIO 34 (ADC) | Thermistor output          | Analog input --- 10-sample average |
| MAX30102 INT  | Leave floating             | Do NOT connect to GND              |

## 2.3 I2C Configuration {#i2c-configuration}

- Clock speed: 100 kHz (I2C_SPEED_STANDARD) --- stable on all breakout boards.

- MAX30102 library NOT used --- raw register access via Wire.beginTransmission / requestFrom.

- Part ID register 0xFF must return 0x15 to confirm sensor presence.

- Full FIFO drain loop on every iteration --- prevents frozen readings.

# 3. ESP32 Firmware {#esp32-firmware}

## 3.1 Sensor Reading Strategy {#sensor-reading-strategy}

|                     |                                                                     |
|---------------------|---------------------------------------------------------------------|
| **Sensor**          | **Strategy**                                                        |
| MAX30102 (IR + Red) | Raw I2C --- drain full FIFO every 100 ms, keep freshest sample      |
| MAX30102 BPM        | Peak detection on filtered IR signal --- valid range 30--200 BPM    |
| MAX30102 SpO2       | Empirical formula: SpO2 = 110 − 25 × (Red / IR)                     |
| MPU6050             | getMotion6() every 100 ms --- magnitude + variance for activity     |
| Thermistor          | 10-sample ADC average + EMA filter --- calibrated with slope/offset |

## 3.2 Activity Detection {#activity-detection}

Activity level is derived from the variance of the accelerometer magnitude vector:

|            |                    |                        |                  |
|------------|--------------------|------------------------|------------------|
| **Level**  | **Variance Range** | **Real-World Meaning** | **Stored Value** |
| Stationary | \< 2,000           | Sitting, lying down    | 0                |
| Low        | 2,000 -- 80,000    | Slow walking           | 1                |
| Moderate   | 80,000 -- 500,000  | Jogging, stairs        | 2                |
| High       | \> 500,000         | Running, jumping       | 3                |

## 3.3 Sleep Mode {#sleep-mode}

- Automatically detected when activity level = 0 AND variance \< 500 for an extended period.

- Stored as sleepMode: true in Realtime DB live node.

- Python ML service applies different insight thresholds during sleep mode.

- Example: BPM of 48 during sleep = normal; BPM of 48 whil\`e awake = caution.

## 3.4 Data Averaging & Aggregation Pipeline {#data-averaging-pipeline}

The system has been updated to aggregate data directly in the Realtime Database to prevent database quota limits:

| Window / Step | Interval / Duration | Action Taken |
| :--- | :--- | :--- |
| **Raw Sensor Readings** | 100 ms | Read on ESP32 (MAX30102 raw I2C, MPU6050, ADC thermistor) |
| **Live Update** | 1 second | ESP32 updates the `live/{userId}` node in Realtime DB |
| **Aggregation Service** | 8 seconds | Python script polls `live`, calculates running average, and updates the `aggregates/{userId}/{date}/{HHMMSS}` (1-minute bucket) |
| **Insights Service** | 8 seconds | Python script polls aggregates, processes rules + ML, and pushes to `insights/{userId}/{date}` |
| **Cleanup Service** | 8 seconds | Python script scans and deletes data older than 24 hours to preserve database limits |

# 6. Python ML & Insights Service {#python-ml-insights-service}

## 6.1 Architecture {#architecture}

- Runs as a standalone background service (local server or Render Background Worker).
- Polls Firebase Realtime Database (`aggregates/` and `live/` nodes) every **8 seconds**.
- Runs rule engine + Random Forest ML model on each new 1-minute bucket.
- Automatically cleans up historical data older than **24 hours** every 8 seconds.
- Writes insight documents back to `/insights/{userId}/{date}`.

## 6.2 Medical Thresholds (Rule Engine) {#medical-thresholds-rule-engine}

|                  |                  |                                       |                    |
|------------------|------------------|---------------------------------------|--------------------|
| **Metric**       | **Normal**       | **Warning**                           | **Danger**         |
| Heart Rate (BPM) | 60 -- 100        | 50--59 or 101--120                    | \< 50 or \> 120    |
| SpO2 (%)         | 95 -- 100        | 92 -- 94                              | \< 92              |
| Temperature (°C) | 36.1 -- 37.2     | 37.3--38.0 (fever) / 35.5--36.0 (low) | \> 38.0 or \< 35.5 |
| Activity         | 0 -- 3 (context) | ---                                   | ---                |

## 6.3 Combined Pattern Rules {#combined-pattern-rules}

- HR \> 100 + SpO2 \< 95 + Activity = 0 → DANGER: serious combination at rest.

- Temp ≥ 38.0 + HR \> 100 + Activity ≤ 1 → WARNING: fever with elevated heart rate.

- Sleep mode + SpO2 \< 94 → WARNING: possible sleep apnea.

- Sleep mode + HR \> 90 → CAUTION: restless sleep or stress.

- Activity = 3 + SpO2 \< 94 → WARNING: oxygen dropping during high activity.

- Activity = 0 + HR \< 55 + Temp \< 36.0 → DANGER: possible shock.

- HR elevated + activity explains it → INFO: expected physiological response.

# 7. Full Data Flow {#full-data-flow}

The complete end-to-end journey of a sensor reading (migrated completely to RTDB):

1.  ESP32 reads sensors every 100 ms (MAX30102 FIFO drain, MPU6050 getMotion6, ADC thermistor).

2.  Raw values filtered with Exponential Moving Average (EMA).

3.  BPM computed via peak detection on filtered IR; SpO2 via Red/IR ratio formula.

4.  Activity level computed from accelerometer magnitude variance.

5.  Sleep mode flag set if activity = 0 and variance \< 500.

6.  **Every 1 second:** Push live snapshot to Firebase Realtime DB under `live/{userId}`.

7.  **Every 8 seconds (Aggregation):** Python service polls `live/{userId}`, aggregates data into a running average for the current minute, and saves to `aggregates/{userId}/{date}/{HHMMSS}`.

8.  **Every 8 seconds (Insights):** Python service polls `aggregates/`, checks for unprocessed buckets, runs rule engine + ML prediction (if confidence >= 70%), and writes insight to `insights/{userId}/{date}/{pushId}`.

9.  **Every 8 seconds (Cleanup):** Python service scans `aggregates` and `insights` nodes and deletes any records older than **24 hours**.

10. **Flutter App Display:** Reads `/live/{userId}` for real-time display, `/aggregates/{userId}` for the 24-hour trend graphs, and `/insights/{userId}` for active medical warnings.

# 9. Agreed Guidelines & Constraints {#agreed-guidelines-constraints}

## 9.1 Hardware Rules {#hardware-rules}

- Never use the MAX30105 Arduino library --- use raw I2C register access only.

- Always verify Part ID register (0xFF = 0x15) before initialising MAX30102.

- I2C clock locked at 100 kHz --- do not use 400 kHz (weak pull-ups on breakout boards).

- FIFO must be fully drained in a while loop every cycle --- single reads cause frozen values.

- Finger detection threshold: filteredIR \> 50,000 (raw MAX30102 values, not library values).

## 9.2 Data Rules {#data-rules}

- Firebase Realtime DB is the sole database (Firestore is completely removed).
- Live node (`/live`) contains instant sensor readings, updated every 1 second by the ESP32.
- Aggregates node (`/aggregates`) contains 1-minute averaged data, updated every 8 seconds by Python.
- Insights node (`/insights`) holds severity-classified events generated by Python.
- Rolling history window is locked at 24 hours; data older than this is automatically pruned by the cleanup routine.
- Date string format: YYYY-MM-DD (from ESP32 NTP time sync).

## 9.3 Medical Rules {#medical-rules}

- All thresholds must match clinically accepted ranges (see Section 6.2).

- Insight messages must be context-aware: the same BPM value means different things at different activity levels.

- Sleep mode uses relaxed heart rate thresholds (low BPM is expected and normal during sleep).

- Combined pattern rules take priority over single-sensor rules.

- The rule engine is the source of truth --- ML model is supplementary.

## 9.4 App Rules {#app-rules}

- Live screen reads exclusively from Realtime DB for minimum latency.

- All graph data comes from Firestore 30min_avg (up to 48 points per day).

- Insights are read-only from the app --- only the Python service writes to insights/.

- Sleep mode screen activates automatically based on the sleepMode flag in Realtime DB.
